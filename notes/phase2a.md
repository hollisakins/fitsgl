# Phase 2a — TypeScript RICE decompression

Standalone, lossless RICE_1 decoder for 32-bit integers, ported from CFITSIO's
`fits_rdecomp` and tested exhaustively against fixtures generated by astropy.
Lives in `fits-pyramid/` (the TS package shared with Phase 2b).

## What was built

- **`src/rice/bitreader.ts`** — `BitReader`, an MSB-first bit reader over a
  `Uint8Array`. One small place owns the byte order (the #1 RICE porting bug).
  `readBits(n)` (0–32), `readUnary()` (RICE fundamental-sequence prefix), with
  explicit bounds checks. Combination uses exact double arithmetic
  (`acc*256+byte`, `Math.floor(acc / POW2[nbits])`) rather than 32-bit bitwise
  ops, so a full 32-bit read (the high-entropy case) returns a correct unsigned
  value with no sign-bit or rounding surprises (accumulator never exceeds ~2³⁹,
  inside the 2⁵³ exact-integer range).
- **`src/rice/rdecomp.ts`** — `rdecompInt`, the port of `fits_rdecomp`. Keeps the
  reference's variable names (`fs`, `nzero`, `lastpix`, `diff`, `fsbits=5`,
  `fsmax=25`, `bbits=32`) and three-branch block structure; the only structural
  change is sourcing bits through `BitReader`.
- **`src/rice/index.ts`** — public `riceDecompress(compressed, nValues, blockSize=32)`
  returning `Int32Array`, with argument validation and descriptive errors.

## Format (32-bit path), as confirmed against the C reference and astropy

- First 4 bytes of the buffer = the first pixel value, **big-endian, uncompressed**
  (seeds `lastpix`).
- Remaining bytes are an MSB-first bitstream of blocks of `nblock` (=32) pixels.
  Each block starts with a 5-bit FS field storing `fs + 1`:
  - field 0 → `fs = -1`: **low-entropy** block, all differences zero → constant run.
  - field 26 (`fsmax+1`) → `fs = 25`: **high-entropy** block, each value coded as
    32 raw bits.
  - otherwise → `fs ∈ 1..24`: **normal Rice**, unary high part + `fs` low bits.
- Each value is a zigzag-mapped difference from the previous pixel; the mapping
  (`even→d>>1`, `odd→~(d>>1)`) and the running sum both wrap mod 2³² by design.
  Storing the unsigned result into `Int32Array` recovers the signed int32 via
  ToInt32.

The reference was obtained verbatim from HEASARC/cfitsio `ricecomp.c`; the byte
format was independently re-derived by probing astropy's low-level
`compress_rice_1_c` codec, and the two agree.

## Fixtures (49, in `test/fixtures/rice_fixtures.json`)

Generated by `generate_fixtures.py` using astropy as the canonical encoder. The
default route writes each int32 array as a single-tile `CompImageHDU` and
extracts the BINTABLE `COMPRESSED_DATA` bytes — byte-for-byte what the Phase 2b
tile fetcher reads, so the fixtures are faithful to the full pipeline (verified:
`ZBITPIX==32`, `ZQUANTIZ` absent → no quantization; identical to the low-level
codec at block size 32). Tests load only the JSON; **no runtime astropy
dependency.** `expected_b64` is little-endian int32; `compressed_b64` is the RICE
stream.

What the inventory covers:
- **Constant / low-entropy blocks** — all-zeros, all-ones, all-`-1`,
  all-`INT32_MAX`/`INT32_MIN`, lengths 1…1024.
- **Single & tiny values** — sign and 4-byte-header coverage.
- **Normal Rice across the FS range** — small randoms, ramps, plus
  `normal_fs_band_{22,23,24}` which deliberately exercise the upper normal-fs
  band (fs 21–24) just below the high-entropy cutoff.
- **High-entropy blocks (fs == FSMAX)** — `alt_large_128`, full-range int32
  randoms, and `rand_full_int32_{33,130,1000}` which produce a **partial final
  high-entropy block** down to a single pixel (block-boundary edge case).
- **Sign extremes** — alternating `INT32_MIN`/`INT32_MAX` (constant ±1 wrap diff).
- **Outliers** — quiet background with `INT32_MIN`/`INT32_MAX` spikes (mixes
  low/normal/high-entropy blocks within one stream).
- **Required lengths** — 32, 33, 100, 1024, 32768, 100000.
- **Non-default block sizes** — 16 and 64 (exercises the `blockSize` parameter,
  via the low-level codec since `CompImageHDU` is fixed at 32).

## Tests (75, ~0.6 s)

- `rice.test.ts`: every fixture decodes **exactly** (no tolerance); edge lengths
  (n=0 → empty, n=1, n=blockSize, n=blockSize+1, n≫blockSize); argument
  validation; and malformed-input handling that throws descriptively —
  too-short header, truncated bitstream, invalid FS/k value, more values
  requested than encoded — while trailing padding bytes do **not** throw.
- `bitreader.test.ts`: the spec widths {1,7,8,9,16,17,32} at various offsets,
  boundary-spanning reads, full 32-bit reads with the high bit set, unary runs
  across byte boundaries, end-of-buffer behavior, plus a 300-trial randomized
  cross-check against an independent reference bit-string reader.

## Adversarial verification

A multi-agent workflow reviewed the port along five lenses (bit order, the three
branches vs the C reference, zigzag/wraparound, malformed-input robustness,
fixture coverage) and adversarially refuted each finding by constructing the
implied input and running the real decoder. **Zero defects survived.** The two
substantive observations were coverage gaps, not bugs (partial high-entropy
final blocks; the fs 21–24 band) — both closed by the fixtures above.

## Decode throughput

Measured with `vitest bench` on Node 23 (V8, same engine as Chrome; browser
numbers should be comparable). Throughput in **decoded output** MB/s:

| fixture | content | output | throughput |
|---|---|---|---|
| `rand_full_int32_32768` | high-entropy (32-bit direct) | 0.13 MB | ~80 MB/s |
| `ramp_100000` | constant-difference ramp | 0.40 MB | ~98 MB/s |
| `rand_small_100000` | small randoms (unary-heavy) | 0.40 MB | ~66 MB/s |

A 256×256 tile (65 536 values, 256 KB) decodes in ~2.6–6 ms depending on content;
the small-random case (closest to real quantized-float visualization tiles) is
the slowest because it does the most unary + small fixed-width reads. A
precomputed power-of-two table (`POW2`) roughly doubled throughput over the
naïve `2 ** nbits`.

## Known limitations

- 32-bit RICE only (`bytepix == 4`). 8- and 16-bit variants are intentionally
  not implemented — astropy's quantized-float output never uses them.
- The decoder validates structure (FS range, buffer length) but, like CFITSIO,
  trusts a well-formed stream past those checks; a stream that is corrupt yet
  structurally plausible decodes to wrong values rather than throwing. This is
  inherent to RICE and is why the fixtures are externally generated.
