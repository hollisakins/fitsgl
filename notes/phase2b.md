# Phase 2b — fpack file parser + tile fetcher

A TypeScript library that parses fpacked FITS files over HTTP range requests,
locates tiles from each file's BINTABLE, and returns decoded `Float32Array`
tiles. Dispatches per file on `ZCMPTYPE`: RICE_1 (lossy, Phase 2a + dequantize)
or GZIP_2 (lossless). Lives in `fits-pyramid/` alongside Phase 2a.

## File structure walkthrough

Both compression types share one on-disk layout; only the heap bytes decode
differently. The byte offsets below were confirmed against real Phase 1 output:

```
byte 0      Primary HDU header   (empty image, NAXIS=0) — 1×2880 block
byte 2880   BINTABLE header      ("COMPRESSED_IMAGE")   — 2×2880 blocks
byte 8640   BINTABLE data unit start
              row table:  NAXIS1 × NAXIS2 bytes  (one row per tile)
byte 8672   heap (= dataStart + THEAP):  variable-length compressed tile bytes
```

- **`fits-header.ts`** — `parseFitsHeader(buf, start)`: walks 80-char cards in
  2880-byte blocks to the `END` card, returns keyword values + the data-unit
  offset. Throws `IncompleteHeaderError` if `END` isn't in the buffer yet (the
  caller fetches more). Typed getters parse ints / quoted strings (trailing-space
  trim, `''` escapes) / logicals.
- **`bintable.ts`** — `parseBinTableLayout`: column byte-offset table from
  `TFORMn`, `THEAP` (default `NAXIS1×NAXIS2`), `PCOUNT`, `heapStart`.
  `readDescriptor` reads a `1P` variable-length descriptor as two big-endian
  int32 `(nElements, heapOffset)` (and `1Q` as int64). `readFloat64BE` for
  ZSCALE/ZZERO.
- **`fpack-file.ts`** — `FpackFile.open` range-fetches ~16 KB, parses both
  headers (growing the fetch if they don't fit), rejects any `ZCMPTYPE` other
  than RICE_1/GZIP_2. `loadTileIndex` fetches the row table once and parses each
  tile's descriptor (+ ZSCALE/ZZERO/ZBLANK for RICE). `getTile(x,y)` looks up
  row `y*nTilesX + x`, range-fetches `heapStart + heapOffset … +nBytes`, and
  dispatches. `httpRangeFetch` verifies HTTP 206 and refuses a 200 (ignored
  Range).
- **`decode-rice.ts`** — `riceDecompress` (Phase 2a) then `int*ZSCALE+ZZERO`,
  with `ZBLANK → NaN`.
- **`decode-gzip2.ts`** — `DecompressionStream('gzip')` → byte unshuffle
  (`unshuffled[i*B+j] = shuffled[j*N+i]`) → big-endian float32. Lossless; NaN bit
  patterns pass through untouched.
- **`tile-source.ts`** — `TileEngine` (manifest + lazy `FpackFile` per level +
  decoded-tile LRU + in-flight de-dup) and `TilePyramid` (public façade).
- **`worker.ts` / `worker-protocol.ts`** — `TilePyramid` runs a `TileEngine`
  inside a Web Worker in the browser; only transferred `Float32Array`s cross to
  the main thread. The worker clones a tile before transferring so the engine's
  cached copy isn't detached. `attachTileWorker` is transport-agnostic and unit
  tested in Node over an in-process scope.
- **`lru.ts`, `manifest.ts`** — small LRU; manifest schema mirroring Phase 1.

## Dispatch logic

`ZCMPTYPE` from the file (never the manifest hint) selects the path:

| keyword | columns present | decode |
|---|---|---|
| `GZIP_2` | `COMPRESSED_DATA` only | gunzip → unshuffle → big-endian f32 (lossless) |
| `RICE_1` | `COMPRESSED_DATA`, `GZIP_COMPRESSED_DATA` (usually empty), `ZSCALE`, `ZZERO` | RICE decode → `int*ZSCALE+ZZERO`, `ZBLANK→NaN` |

`ZBLANK` is read from a per-row column if present, else the header keyword, else
`NaN` (so the `v === zblank` blank test is a guaranteed no-op when a file
declares no blanks — a real int can never equal NaN).

**Partial edge tiles:** astropy stores tiles on the high-index edge of a
non-256-divisible image at their actual smaller size. `getTile` computes each
tile's true pixel count via `tileDims` (`min(ztile, znaxis − tile*ztile)`) and
decodes to that length, rather than assuming 256×256. (The committed 512×512
fixtures are all full tiles; the partial path is validated separately — see
verification below.)

**Per-tile GZIP fallback:** the `GZIP_COMPRESSED_DATA` column is present in RICE
files but empty for normally-compressed tiles. An empty cell is ignored; a
non-empty one (a tile fpack stored gzipped instead of RICE) is rejected with a
clear error rather than mis-decoded.

## Test coverage (per path)

48 Phase 2b tests over **real fpack files** generated by Phase 1 from a 512×512
synthetic mosaic (committed: a GZIP_2 z=0 file with 2×2 tiles and a RICE_1 z=1
file with 1 tile; regen via `test/fixtures/generate_pyramid_fixtures.py`).

- `fits-header.test` — primary + BINTABLE structure, ZCMPTYPE/ZBLANK/ZTILE
  extraction, data-start offsets, trailing-space trim, `IncompleteHeaderError`.
- `bintable.test` — column layout + offsets, ZSCALE/ZZERO present for RICE and
  **absent for GZIP_2**, descriptor (nBytes, offset) sequential and summing to
  PCOUNT, float64 ZSCALE/ZZERO.
- `decode-gzip2.test` — sample tile decodes **bit-exact** to the native image
  (NaN positions included); gunzip size guard.
- `decode-rice.test` — sample tile decodes **exactly** to astropy's own decode
  (a tighter check than q=16 vs the original); ZBLANK→NaN count matches.
- `fpack-file.test` — open metadata for both types; **rejects unknown ZCMPTYPE**
  and **lossless/integer RICE** (no ZSCALE/ZZERO); header-growth path with a tiny
  initial fetch; tile index descriptors; all four GZIP_2 tiles lossless vs native;
  RICE tile vs astropy; out-of-range throws; httpRangeFetch 206/200/network
  behavior.
- `tile-source.test` — manifest load; GZIP_2 and RICE_1 `getTile`; **end-to-end
  lossless guarantee** (every z=0 tile reconstructs the native 512×512 exactly);
  LRU eviction; concurrent identical requests dedupe to one fetch+decode;
  cache-hit no-refetch; **worker mode** through an in-process worker (correct
  tiles, main-side dedup, `destroy()` terminates + rejects, worker errors
  surface, **init-failure terminates the worker** so a failed load can't leak it).
- `lru.test` — eviction, recency refresh, capacity validation.

No runtime astropy dependency: ground truth (`native.bin`, `z1_decoded.bin`,
sample tile bytes) is committed.

## Adversarial verification

A multi-agent workflow reviewed Phase 2b along five lenses (the GZIP_2 byte
order/unshuffle, BINTABLE + heap addressing, header parsing, dispatch/dequant,
and the engine/worker lifecycle), each finding adversarially refuted by
generating a *fresh* fpack file — including **non-256-divisible mosaics** to
exercise partial edge tiles, which the committed 512×512 fixtures don't — and
comparing the TS decode to astropy tile by tile. The core decode paths (byte
order, unshuffle, heap math, partial-tile dimensions) all verified correct. It
surfaced two robustness defects, both now fixed and regression-tested:

1. **Silent all-NaN on lossless/integer RICE** — a RICE file lacking ZSCALE/ZZERO
   columns decoded to a fully-NaN tile with no error. `FpackFile` now rejects such
   files at open with a clear message (structural column-presence check, so a
   legitimately all-blank tile is unaffected).
2. **Worker leak on init failure** — a failed `TilePyramid.load` in worker mode
   (e.g. a 404 manifest) rejected without terminating the spawned worker, which
   the caller could no longer reach. `load` now tears the worker down before
   rethrowing.

## Build

The library builds with `tsc -p tsconfig.build.json` (not a bundler), preserving
the module tree under `dist/` so the worker URL
`new URL('../worker.js', import.meta.url)` in `dist/fpack/tile-source.js` resolves
to `dist/worker.js` in a consumer's bundler. `src/` uses no Node APIs; `@types/node`
is scoped to the tests via an empty `types` array.

## Observed per-tile decode latency (256×256, Node 23 / V8)

| path | per tile | relative |
|---|---|---|
| GZIP_2 (gunzip + unshuffle + BE f32) | ~1.4 ms | 1× |
| RICE_1 (RICE decode + dequantize) | ~3.9 ms | 2.8× slower |

RICE is slower because it bit-decodes every quantized value, whereas GZIP_2 is a
streamed gunzip plus two linear byte passes. Both run in a worker off the main
thread.

## Known limitations

- RICE_1 and GZIP_2 only; any other `ZCMPTYPE` is rejected with a clear error.
- `1P` (32-bit) descriptors are the norm; `1Q` (64-bit) is handled but untested
  against a real >2 GB heap.
- A per-tile GZIP fallback (`GZIP_COMPRESSED_DATA` non-empty) is detected and
  rejected, not decoded — astropy emits it only in unusual cases our pipeline
  doesn't produce.
- The browser worker path uses `httpRangeFetch` + global `fetch`; custom fetchers
  (used by tests) only apply to the inline engine, since closures can't cross a
  real worker boundary.
